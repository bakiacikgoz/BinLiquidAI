4) Approval Continuation (Resume Semantics)
====================================================================
4.1 What was missing before
- Team runs correctly escalated/blocked on approval-required gates.
- But continuation after approval required external/manual restart semantics.

4.2 What is implemented now
- New CLI command:
  - `binliquid team resume --spec <spec> --job-id <blocked_job> --root-dir <jobs_dir> [--resume-job-id ...]`
- Resume flow:
  1. Load source job status/events.
  2. Discover `approval_requested` events.
  3. Resolve approval tickets from approval store.
  4. Keep only `approved|executed` tickets.
  5. Build task-target override map (`task|handoff|memory_write`).
  6. Start new team run for same request/case with overrides.
- Supervisor/orchestrator glue:
  - Task-level override passes via session context `governance_approval_id`.
  - Handoff/memory write evaluators accept `override_approval_id`.
  - Governance runtime returns ALLOW on approved override with `matched_rule_path=approval_override`.

4.3 Coverage
- `governance/runtime.py`: task + handoff + memory_write override path
- `team/supervisor.py`: emits `approval_resolved` and applies per-task target override
- `team/handoff.py`, `team/memory_scope.py`: override plumbed
- `cli.py`: `team resume` implementation + override collector
- Tests cover explicit override behavior and full CLI resume flow.

5) Policy and Profile Hardening (Regulated Baseline)
====================================================================
5.1 New restricted policy profile
- Runtime profile: `config/restricted.toml`
- Governance policy: `config/policies/restricted.toml`

5.2 Intent
- Reduce mismatch between regulated positioning and permissive defaults.
- Introduce least-privilege defaults for handoff/memory/tooling around high-risk paths.

5.3 Notable restricted policy behavior
- `research`, `code`, `mixed` routes are approval-gated.
- Python tool invocation requires approval.
- Transitions into `Execution Agent` require approval.
- Case/team memory writes by `Execution Agent` require approval.
- Session-private and selected case-private writes are allowed.
- Unmatched handoff/memory routes remain deny-by-default by evaluator fallback.

5.4 Team template support

  with role set:
  - Intake Agent

  - Policy/Compliance Agent
  - Execution Agent
  - Reviewer/QA Agent

6) SQLite Concurrency Hardening
====================================================================
6.1 Memory store (`PersistentMemoryStore`)
- Connection uses `check_same_thread=False`.
- Internal `RLock` serializes operations.
- Connection pragmas:
  - `PRAGMA journal_mode=WAL`
  - `PRAGMA synchronous=NORMAL`
  - `PRAGMA busy_timeout=5000`

6.2 Team checkpoint store (`TeamCheckpointStore`)
- Thread-safe access via `RLock`.
- `check_same_thread=False`.
- Same WAL/busy_timeout pragmas applied.

6.3 Approval store (`ApprovalStore`)
- Per-connection pragmas include WAL/synchronous/busy_timeout.

6.4 Stress/regression tests
- `tests/test_memory_concurrency.py`

8) Audit Integrity and Signature
====================================================================
8.1 Existing integrity model
- Per-job envelope includes:
  - `integrity.prev_hash`
  - `integrity.hash`
  - `integrity.signature`
- Hash chain head persisted under team artifact root parent.

8.2 New signature support
- If `BINLIQUID_AUDIT_SIGNING_KEY` is set:
  - signature is computed as HMAC-SHA256 over current envelope hash.
- If unset:
  - signature remains `null` (previous behavior).
