4.1 Core runtime (existing, preserved)

4.2 New Team Runtime layer
New package files:
- binliquid/team/models.py
- binliquid/team/scheduler.py
- binliquid/team/supervisor.py
- binliquid/team/handoff.py
- binliquid/team/memory_scope.py
- binliquid/team/artifacts.py
- binliquid/team/replay.py
- binliquid/team/checkpoint_store.py
- binliquid/team/__init__.py

Responsibilities:
- models.py: team/job/task/handoff/event/envelope typed contracts
- scheduler.py: parallel dependency-aware task scheduling
- supervisor.py: team execution loop, handoff and memory governance, job status resolution
- handoff.py: inter-task payload governance evaluation and payload hashing
- memory_scope.py: memory write governance + scoped write helper
- artifacts.py: team job files + audit envelope hash chaining
- replay.py: job status/events/envelope reconstruction
- checkpoint_store.py: SQLite checkpoint state persistence per job

4.3 Team runtime execution flow (implemented)
1) team run command resolves runtime config and team spec.
2) TeamSupervisor instantiates job context (case_id/job_id/team_id).
3) Tasks are materialized from spec (or deterministic auto-decomposition fallback).
4) ParallelScheduler executes runnable tasks by dependency readiness.
5) Each task invokes orchestrator; chat tasks can use fast path when safe.
6) Inter-task handoff passes governance evaluation + redaction.
7) Scoped memory write passes governance evaluation.
8) Task/job metrics and events are persisted.
9) Team audit envelope is generated with hash-chain integrity fields.
10) status/log/replay/artifacts commands operate on persisted job data.

11.2 Thread safety defect in parallel team run

- SQLite ProgrammingError in task worker thread.

- check_same_thread=False + RLock guarding DB operations.
